## 一、RAG 智能对话功能

### 1.1 功能概述

用户在前端输入问题 → 后端 Agent 接收问题 → Agent 自主决定是否调用工具（知识库检索、查告警、查时间等）→ 返回答案。

支持两种模式：**一次性返回**（`/api/chat`）和**流式逐字返回**（`/api/chat_stream`）。

### 1.2 请求入口

| 接口 | 方法 | 路径 | 核心文件 |
|------|------|------|----------|
| 快速问答 | POST | `/api/chat` | `app/api/chat.py:19` |
| 流式对话 | POST | `/api/chat_stream` | `app/api/chat.py:70` |
| 清空会话 | POST | `/api/chat/clear` | `app/api/chat.py:173` |
| 查询历史 | GET | `/api/chat/session/{id}` | `app/api/chat.py:198` |

请求体模型定义在 `app/models/request.py:9`：
```python
class ChatRequest(BaseModel):
    id: str       # 会话 ID，用于多轮对话记忆
    question: str # 用户问题
```

### 1.5 Agent 内部执行机制

核心文件：`app/services/rag_agent_service.py`

```python
# 第 95 行：创建 LLM 模型
self.model = ChatQwen(
    model="qwen-max",
    api_key=config.dashscope_api_key,
    temperature=0.7,
    streaming=True,
)

# 第 103 行：定义本地工具集
self.tools = list(DEFAULT_LOCAL_AGENT_TOOLS)
# 来自 app/tools/__init__.py:8：
#   1. retrieve_knowledge    — 从 Milvus 知识库检索
#   2. get_current_time       — 获取当前时间
#   3. query_prometheus_alerts — 查 Prometheus 告警

# 第 113 行：确认 Agent 未初始化时，异步初始化
async def _initialize_agent(self):
    # 第 130-138 行：加载 MCP 工具（日志查询、监控数据）
    mcp_client = await get_mcp_client_with_retry()
    mcp_tools, err = await load_mcp_tools_safe(mcp_client)

    # 第 141 行：合并本地工具 + MCP 工具
    all_tools = self.tools + self.mcp_tools

    # 第 143-147 行：创建 LangGraph Agent
    self.agent = create_agent(
        self.model,
        tools=all_tools,
        checkpointer=self.checkpointer,  # MemorySaver，内存级会话持久化
    )
```

**关键点**：`create_agent()` 是 LangChain 提供的高级 API，它内部会自动处理 ReAct 循环（思考→行动→观察→思考→...），直到 LLM 认为问题已解决，不需要手写循环逻辑。

## 二、AIOps 智能故障诊断功能

### 2.1 功能概述

这是项目的**核心亮点**。用户（或定时任务）触发诊断后，系统自动执行完整的故障排查流程：制定计划 → 逐步执行 → 动态调整 → 生成报告。整个过程通过 SSE 流式推送到前端。

诊断的默认任务定义在 `app/services/aiops_service.py:174-246`，是一个固定的 AIOps 任务描述，要求 LLM 生成结构化的 Markdown 诊断报告。

### 2.2 请求入口

| 接口 | 方法 | 路径 | 核心文件 |
|------|------|------|----------|
| AIOps 诊断 | POST | `/api/aiops` | `app/api/aiops.py:16` |

请求体：`app/models/aiops.py:9`
```python
class AIOpsRequest(BaseModel):
    session_id: Optional[str] = "default"
```

### 2.3 整体架构：Plan-Execute-Replan 状态机

核心文件：`app/services/aiops_service.py`

这是一个基于 **LangGraph StateGraph** 的三节点工作流：

```
入口
  │
  ▼
┌──────────┐     ┌──────────┐     ┌───────────┐
│ PLANNER  │────▶│ EXECUTOR │────▶│ REPLANNER │
│ 制定计划 │     │ 执行步骤 │     │ 评估+决策 │
└──────────┘     └──────────┘     └─────┬─────┘
                                        │
                          ┌─────────────┼─────────────┐
                          ▼             ▼             ▼
                      continue       replan       respond
                      (继续执行)    (调整计划)    (生成报告)
                          │             │             │
                          ▼             ▼             ▼
                      EXECUTOR      EXECUTOR        END
```

状态定义在 `app/agent/aiops/state.py:10`：
```python
class PlanExecuteState(TypedDict):
    input: str                              # 用户输入/诊断任务
    plan: List[str]                         # 待执行的步骤列表
    past_steps: Annotated[List[tuple], operator.add]  # 已完成步骤 [(步骤, 结果)]
    response: str                           # 最终响应/报告
```

图构建代码在 `app/services/aiops_service.py:29-79`：
```python
def _build_graph(self):
    workflow = StateGraph(PlanExecuteState)
    workflow.add_node("planner", planner)       # app/agent/aiops/planner.py
    workflow.add_node("executor", executor)     # app/agent/aiops/executor.py
    workflow.add_node("replanner", replanner)   # app/agent/aiops/replanner.py
    workflow.set_entry_point("planner")
    workflow.add_edge("planner", "executor")
    workflow.add_edge("executor", "replanner")
    # replanner 根据 should_continue() 条件路由
    workflow.add_conditional_edges("replanner", should_continue, {
        "executor": "executor",  # 继续执行
        END: END                 # 生成报告，结束
    })
```

### 2.4 三节点详解

#### 2.4.1 Planner（规划器）

**文件**：`app/agent/aiops/planner.py:63`

**执行流程**：

```
planner(state)
  │
  ├── [1] 检索知识库                                    ← planner.py:79-90
  │       retrieve_knowledge.ainvoke({"query": input_text})
  │       从 aiops-docs/ 中检索相关运维经验文档
  │       如：查 CPU 告警时，会检索到 cpu_high_usage.md
  │
  ├── [2] 获取所有可用工具                               ← planner.py:93-101
  │       本地工具 (DEFAULT_LOCAL_AGENT_TOOLS)
  │       + MCP 工具 (日志查询、监控数据)
  │
  ├── [3] 构建 Prompt，调用 LLM 生成计划                  ← planner.py:122-135
  │       prompt 包含：工具列表 + 经验文档 + 用户任务
  │       LLM 输出 Plan 对象:
  │         class Plan(BaseModel):
  │             steps: List[str]  # 如 ["步骤1: 获取当前时间",
  │                               #      "步骤2: 查Prometheus告警", ...]
  │
  └── [4] 返回 {"plan": ["步骤1", "步骤2", ...]}          ← planner.py:148
```

**关键 Prompt**（`planner.py:28-60`）：
- 要求 LLM 将复杂任务分解为逻辑独立的步骤
- 每个步骤明确使用哪些工具、提供什么参数
- 如果有经验文档，参考其中的方法和步骤

**返回格式**（`planner.py:20-24`）：
```python
class Plan(BaseModel):
    steps: List[str]  # 4-6 个诊断步骤
```

#### 2.4.2 Executor（执行器）

**文件**：`app/agent/aiops/executor.py:18`

**执行流程**：

```
executor(state)
  │
  ├── [1] 取出 plan[0]（当前步骤）                        ← executor.py:34
  │
  ├── [2] 获取工具并绑定到 LLM                            ← executor.py:39-55
  │       llm.bind_tools(all_tools)
  │       创建 ToolNode(all_tools) 用于自动执行工具调用
  │
  ├── [3] LLM 接收步骤描述，决定是否调用工具               ← executor.py:79
  │       llm_response = await llm_with_tools.ainvoke(messages)
  │
  ├── [4] 如果有 tool_calls，执行工具并获取结果             ← executor.py:83-93
  │       tool_messages = await tool_node.ainvoke(...)
  │       将工具结果再送回 LLM 生成自然语言描述
  │
  └── [5] 返回状态更新                                    ← executor.py:102-104
          {
              "plan": plan[1:],              # 移除已执行的步骤
              "past_steps": [(task, result)] # 追加执行历史
          }
```

**关键点**：Executor 每次只执行**一个步骤**（`plan[0]`），执行完就移除。它使用 LangGraph 的 `ToolNode` 自动处理工具调用的循环（LLM 调用工具 → 工具返回 → LLM 总结）。

#### 2.4.3 Replanner（重规划器）

**文件**：`app/agent/aiops/replanner.py:111`

**执行流程**：

```
replanner(state)
  │
  ├── [1] 保护逻辑：已执行 ≥ 8 步 → 强制生成响应            ← replanner.py:130-138
  │
  ├── [2] 获取当前工具列表                                  ← replanner.py:141-157
  │
  ├── [3] LLM 决策（三选一）                                ← replanner.py:176-189
  │       使用 Act 模型进行结构化输出：
  │         class Act(BaseModel):
  │             action: str      # "continue" | "replan" | "respond"
  │             new_steps: List[str]  # 仅 action="replan" 时需要
  │
  ├── [4] 根据决策执行不同分支                              ← replanner.py:202-234
  │       ├── respond → _generate_response()  生成最终报告
  │       ├── replan  → 返回 {"plan": new_steps}  替换剩余计划
  │       │             但有约束：new_steps ≤ 原剩余步骤数
  │       │             且已执行 ≥ 5 步时禁止 replan
  │       └── continue → 返回 {}  不修改状态，继续原计划
  │
  └── [5] 如无剩余计划 → _generate_response()               ← replanner.py:239
```

**_generate_response() 详细流程**（`replanner.py:242-291`）：

```
_generate_response(state, llm)
  │
  ├── 格式化所有已执行步骤和结果
  │       execution_history = "\n\n".join([
  │           f"### 步骤: {step}\n**结果:**\n{result}"
  │           for step, result in past_steps
  │       ])
  │
  ├── 调用 LLM 生成结构化响应
  │       response_prompt | llm.with_structured_output(Response)
  │       class Response(BaseModel):
  │           response: str   # Markdown 格式的报告
  │
  └── 返回 {"response": "## 告警分析报告\n..."}
```

**决策优先级**（Prompt 中的口诀，`replanner.py:83`）：
> "优先结束 > 保持不变 > 调整计划"
> "信息足够就响应，不要追求完美"

**安全约束**：
- 已执行 ≥ 8 步 → 强制结束
- 已执行 ≥ 5 步 → 禁止重新规划
- 新步骤数不能超过当前剩余步骤数

### 2.5 流式事件类型

诊断过程通过 SSE 推送以下事件类型（`app/services/aiops_service.py:264-337`）：

| 事件 type | stage | 含义 | 产生节点 |
|-----------|-------|------|----------|
| `plan` | `plan_created` | 诊断计划已制定，含步骤列表 | Planner |
| `step_complete` | `step_executed` | 单个步骤执行完成 | Executor |
| `status` | `executor` / `replanner` | 执行中 / 评估中 | Executor / Replanner |
| `report` | `final_report` | 最终报告已生成 | Replanner |
| `complete` | `diagnosis_complete` | 诊断流程完全结束 | AIOpsService |
| `error` | `error` | 异常错误 | 任意 |

### 2.6 前端调用 AIOps 的入口

在 `static/app.js` 中，用户点击侧边栏的"智能运维与诊断工具"按钮触发：

```javascript

eventSource.onmessage = (event) => {
    const data = JSON.parse(event.data);
    if (data.type === 'plan')       { /* 展示诊断计划 */ }
```


### 3.3 MCP 协议层

**核心文件**：`app/agent/mcp_client.py`

MCP（Model Context Protocol）是本项目的**关键设计**——它把外部数据源（日志系统、监控系统）封装为 LLM 可以调用的"工具"。

```
Agent (LLM)                         MCP Servers
    │                                    │
    │  调用工具 "search_log"             │
    ├──────────────────────────────────▶│
    │  MCPToolCallRequest               │  cls_server.py (端口 8003)
    │  {name, args, server_name}        │
    │                                    │
    │  返回 CallToolResult               │
    │◀──────────────────────────────────┤
    │  {content, isError}               │
```

**MCP 客户端管理**（`mcp_client.py:16-211`）：
- 全局单例模式，整个应用共享一个 `MultiServerMCPClient`
- 自动加载两个 MCP 服务器配置（来自 `app/config.py:57-68`）
- 内置重试拦截器（`retry_interceptor`），最多重试 3 次，指数退避
- URL 与 transport 不匹配时会给出修正建议（`suggest_mcp_transport`）

```
app/tools/__init__.py
```
