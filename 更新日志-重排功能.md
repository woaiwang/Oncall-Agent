# 更新日志 — RAG 文档重排（Reranking）功能

> 日期：2026-06-22
> 功能：向量检索后增加语义重排，提升 RAG 回答质量

---

## 一、为什么要加重排？

### 问题

当前 RAG 流程：
```
用户问题 → Embedding → Milvus 向量检索(L2距离) → Top-3文档 → 喂给LLM
```

**向量检索的局限**：Milvus 用 L2 距离（欧氏距离）比较 query 向量和 doc 向量的相似度。但「向量空间中的距离近」≠「语义上真正相关」。举例：

| 用户问题 | Milvus L2返回的"相关"文档 | 真正应该返回的 |
|----------|--------------------------|--------------|
| "CPU使用率过高" | 磁盘问题文档（因为"使用率"这个词向量被带偏） | CPU问题文档 |
| "服务响应慢" | 内存泄漏文档（都可能提到"性能"但语义不同） | 响应慢/超时文档 |

核心原因：Bi-Encoder（Embedding 模型）把 query 和 doc **各自独立**编码为向量，丢失了**query 和 doc 之间的细粒度语义交互**。

### 解决方案：Cross-Encoder 重排

```
                        向量检索（粗排）               重排（精排）
用户问题  ──→  Embedding  ──→ Milvus Top-9  ──→ gte-rerank 打分排序 ──→ Top-3 ──→ LLM
          query+doc分别编码      L2距离近似匹配      query+doc联合编码
          速度快，精度低                            速度慢但精度高
```

**Cross-Encoder** 把 (query, doc) 拼接后一起编码，模型能看到两者的完整语义交互，判断出的相关性远比 L2 距离准确。

### 类比理解

| 阶段 | 类比 | 做什么 | 精度 | 速度 |
|------|------|--------|------|------|
| 向量检索（粗排） | 简历筛选 | 从 1000 份简历中筛出 10 份 | 低 | 快 |
| 重排（精排） | 面试 | 对 10 份简历逐一深入评估，选出 3 份 | 高 | 慢 |

## 二、技术方案

### 选型：阿里云百炼 gte-rerank

**为什么不选本地 Cross-Encoder（如 BAAI/bge-reranker-v2-m3）？**

| 维度 | 本地 Cross-Encoder | 百炼 gte-rerank API |
|------|-------------------|-------------------|
| 模型文件 | ~2GB 下载 | 无 |
| 依赖 | sentence-transformers + torch (~3GB) | 已有 dashscope |
| 推理 | 占用 CPU/GPU | 阿里云端推理 |
| 延迟 | ~50-200ms（取决于硬件） | ~200-500ms（网络+推理） |
| 费用 | 免费（消耗本机算力） | API 调用费用 |
| 维护 | 需管理模型版本 | 阿里维护升级 |

选择百炼 API 的理由：
- 项目已深度依赖 DashScope（LLM + Embedding 都是百炼），保持一致
- 零新增依赖
- 模型由阿里持续升级

### 百炼重排 API

```python
from dashscope import TextReRank

response = TextReRank.call(
    model="gte-rerank",         # 重排模型
    query="CPU使用率过高怎么办",  # 用户问题
    documents=[                 # 候选文档列表
        "文档1内容...",
        "文档2内容...",
        ...
    ],
    top_n=3,                    # 返回前3个
    return_documents=False,     # 不返回原文（用index映射）
)
# response.output.results = [
#     {"index": 2, "relevance_score": 0.98},
#     {"index": 0, "relevance_score": 0.76},
#     {"index": 5, "relevance_score": 0.45},
# ]
```

### 3.1 `app/services/reranker_service.py`（新建）

**设计决策**：
- **不降级**：如果重排出错，直接抛异常。上游 `retrieve_knowledge` 的 `try/except` 统一捕获，把错误文本返回给 LLM，LLM 能看到"重排失败"并据此调整回答策略
- **全局单例**：和 `vector_embedding_service`、`vector_store_manager` 一致的模式
- **短路优化**：文档数 ≤ top_n 时直接返回，不调 API（省一次网络调用）

### 3.2 `app/config.py`（修改）

新增 4 个配置项：

```python
# Reranker 重排配置（阿里云百炼 gte-rerank 模型）
rerank_enabled: bool = True         # 是否启用重排
rerank_model: str = "gte-rerank"    # 百炼重排模型
rerank_top_n: int = 3               # 重排后保留的文档数
rerank_retrieval_k: int = 9         # 初检从Milvus获取的文档数（候选池）
```

**为什么 `retrieval_k=9` 而 `top_n=3`？**

| 阶段 | 数量 | 原因 |
|------|------|------|
| 向量检索 | 9 | 多拉一些候选，减少漏掉相关文档的概率 |
| 重排 | 3 | 从9个候选中挑出最相关的3个 |
| 喂给LLM | 3 | 控制在LLM上下文窗口内，太多会稀释注意力 |

### 3.3 `app/tools/knowledge_tool.py`（修改）

**修改前**：
```python
retriever = vector_store.as_retriever(search_kwargs={"k": config.rag_top_k})  # k=3
docs = retriever.invoke(query)
context = format_docs(docs)
```

**修改后**：
```python
# 重排开启时多检索一些做候选池
if reranker_service.enabled:
    retrieval_k = config.rerank_retrieval_k  # 9
else:
    retrieval_k = config.rag_top_k           # 3

retriever = vector_store.as_retriever(search_kwargs={"k": retrieval_k})
docs = retriever.invoke(query)

# 重排（仅当候选数 > 保留数时才调API）
if reranker_service.enabled and len(docs) > config.rerank_top_n:
    docs = reranker_service.rerank(query, docs)

context = format_docs(docs)
```

**受益范围**：`retrieve_knowledge` 被两个功能共用：
- **RAG 对话**（`/api/chat` + `/api/chat_stream`）：Agent 调用知识检索工具时自动重排
- **AIOps 诊断**（`/api/aiops`）：Planner 制定计划前检索经验文档时自动重排

## 四、配置方法

### 在 `.env` 中添加（可选，都有默认值）

```bash
# 重排配置
RERANK_ENABLED=True          # 开启重排
RERANK_MODEL=gte-rerank      # 百炼重排模型
RERANK_TOP_N=3               # 重排后保留几个文档
RERANK_RETRIEVAL_K=9         # 初检从Milvus多拉几个候选
```

### 关闭重排

```bash
RERANK_ENABLED=False
```
关闭后行为回退到原来的：Milvus 直接检索 `RAG_TOP_K`（默认3）个文档，不做重排。

## 六、架构影响

### 延迟影响

| 阶段 | 原来 | 现在 | 增量 |
|------|------|------|------|
| 向量检索 | ~50ms (3条) | ~100ms (9条) | +50ms |
| 重排 | 无 | ~300ms (1次API调用) | +300ms |
| LLM推理 | ~2-5s | ~2-5s | 0 |
| **总计** | **~2-5s** | **~2.4-5.4s** | **+~350ms** |

重排增加约 350ms 延迟，换取更好的文档质量。如果对延迟敏感，可以调小 `RERANK_RETRIEVAL_K`。

### 费用影响

百炼 gte-rerank 按 token 计费。每次重排消耗约 2000-5000 tokens（取决于文档长度），在可接受范围内。

## 七、总结

| 维度 | 改动前 | 改动后 |
|------|--------|--------|
| 检索流程 | Embedding → Milvus L2 Top-3 | Embedding → Milvus Top-9 → 百炼重排 Top-3 |
| 排序依据 | 向量空间距离 | 语义相关性分数 |
| 依赖 | dashscope 已有 | 无新增 |
| 配置 | 无 | 4个可选配置项 |
| 降级策略 | 无（不适用） | 失败直接报错，不降级 |
| 受益功能 | — | RAG对话 + AIOps诊断 |

核心思想一句话：**向量检索做粗筛，Cross-Encoder 做精选。**
