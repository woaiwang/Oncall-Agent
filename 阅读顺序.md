## 阅读前准备

**先搞清楚三个问题再开始读代码：**

1. **这个项目解决什么问题？** — 企业级智能运维助手，让运维人员通过自然语言对话即可排查故障，而非手动翻日志/看监控面板。
2. **核心能力是什么？** — 两个能力：RAG 知识库问答（"这个问题在文档里怎么说的？"）+ AIOps 自动故障诊断（"系统有没有告警？根因是什么？怎么修？"）。
3. **核心竞争力在哪里？** — Plan-Execute-Replan 模式（不是简单的 RAG，而是 LLM 自主制定排查计划、调用工具执行、动态调整计划）；MCP 协议解耦工具和 Agent（新增一个数据源不需要改 Agent 代码）。

### rag_agent_service.py 精读要点

这个文件是整个项目最重要的文件之一，共约 419 行，需要逐段理解：

#### 3a. AgentState 和消息修剪（第 36-78 行）

```python
class AgentState(TypedDict):
    messages: Annotated[Sequence[BaseMessage], add_messages]
```

- `add_messages` 是 LangGraph 的 reducer，表示新消息会追加到已有消息列表而不是覆盖
- `trim_messages_middleware`：当消息超过 7 条时，保留首条 SystemMessage + 最近 6 条
- **面试要点**：为什么要修剪？因为 LLM 有上下文窗口限制（qwen-max 约 32K tokens），不修剪会超出限制导致报错或截断

#### 3b. `__init__` 方法（第 84-115 行）

- 模型：`ChatQwen(model="qwen-max", streaming=True)` — 直接使用 langchain_qwq 的原生集成
- 工具：`DEFAULT_LOCAL_AGENT_TOOLS`（来自 `app/tools/__init__.py`）= 知识检索 + 时间 + Prometheus 告警
- 检查点：`MemorySaver()` — LangGraph 的内存级状态持久化，用于多轮对话
- Agent 延迟初始化（`_agent_initialized = False`），因为 MCP 工具需要异步加载
- **面试要点**：为什么用 MemorySaver 而不是数据库？因为会话数据不需要持久化到磁盘，重启后丢失可接受。生产环境可换成 SqliteSaver 或 PostgresSaver

#### 3c. `_initialize_agent` 方法（第 117-154 行）

- 先加载 MCP 工具（`get_mcp_client_with_retry()` + `load_mcp_tools_safe()`），失败不阻塞，降级为仅本地工具
- 合并本地工具 + MCP 工具 = 完整的工具池
- 调用 `create_agent(model, tools, checkpointer)` — **这是 LangChain 的高级 API，内部自动实现 ReAct 循环**
- **面试要点**：`create_agent()` 做了什么？它内部创建了一个 LangGraph 状态图，节点包括：LLM 思考节点 + ToolNode 执行节点，循环直到 LLM 不再调用工具

#### 3d. `query` 方法（第 186-249 行，非流式）

- 构建 `[SystemMessage, HumanMessage]`
- `self.agent.ainvoke(input, config)` — 一次性调用，等待全部完成
- 从 `result["messages"][-1]` 提取最终答案
- `config` 中的 `thread_id` 就是 `session_id`，用于多轮对话的记忆隔离

#### 3e. `query_stream` 方法（第 251-319 行，流式）

- `self.agent.astream()` with `stream_mode="messages"` — 每产生一个 token 就 yield
- 过滤出 `AIMessage` / `AIMessageChunk` 类型，从中提取 `content_blocks` 里的文本
- **面试要点**：流式和非流式的区别？流式用 `astream`（async generator），非流式用 `ainvoke`。流式对用户体验更好（逐字显示），但实现更复杂

#### 3f. 会话管理（第 321-405 行）

- `get_session_history`：从 MemorySaver 的 checkpoint 中读取指定 thread_id 的消息，过滤掉 SystemMessage
- `clear_session`：调用 `checkpointer.delete_thread(session_id)`

### aiops_service.py 精读要点

#### 3a. 状态图的构建（第 29-79 行）

```python
workflow = StateGraph(PlanExecuteState)
workflow.add_node("planner", planner)
workflow.add_node("executor", executor)
workflow.add_node("replanner", replanner)
workflow.set_entry_point("planner")
workflow.add_edge("planner", "executor")
workflow.add_edge("executor", "replanner")
workflow.add_conditional_edges("replanner", should_continue, {
    "executor": "executor",
    END: END
})
```

- **面试要点**：这是一个三节点的有限状态机。Planner → Executor → Replanner → (Executor | END)。`should_continue` 函数是条件路由的核心：如果有 response 就结束，如果还有 plan 就继续，都没有也结束。

#### 3b. `execute` vs `diagnose` 的区别（第 81-262 行）

- `execute`：通用 Plan-Execute-Replan，接受任意 `user_input`
- `diagnose`：封装了固定的 AIOps 诊断任务描述（硬编码的 Markdown 报告模板），然后调用 `execute`
- **面试要点**：这是一个好的设计——通用框架 + 固定模板的封装，AIOps 是 Plan-Execute-Replan 的一个具体应用场景

### replanner.py 精读要点（第 111-305 行）

这是整个项目的**最难点**，也是面试中最可能被问到的部分。

#### 决策逻辑

LLM 每次做三选一决策（用 `Act` 这个 Pydantic 模型做结构化输出）：

1. **`respond`** — 信息足够，生成最终报告（最高优先级）
2. **`continue`** — 当前计划合理，继续执行（次优先级）
3. **`replan`** — 当前计划有问题，调整步骤（最低优先级）

#### 安全约束（防止死循环）

- 已执行 ≥ 8 步 → 强制 `respond`（硬编码保护）
- 已执行 ≥ 5 步 → 禁止 `replan`（只能 `respond` 或 `continue`）
- 新步骤数 ≤ 原剩余步骤数（防止步骤膨胀）
- Prompt 口诀："优先结束 > 保持不变 > 调整计划"、"信息足够就响应，不要追求完美"

#### `_generate_response` 方法

- 格式化所有 `past_steps` 的执行历史
- 调用 LLM 生成结构化响应（用 `with_structured_output(Response)`）
- 如果 LLM 调用失败，生成降级报告（包含原始任务和已执行步骤的摘要）

**面试要点**：为什么要禁止超过 5 步的 replan？因为：
1. 防止 LLM 陷入无限重规划循环（安全和成本考虑）
2. 已执行 5 步通常已收集足够信息
3. 强制 LLM 基于已有信息得出结论，而非继续探索

async def retry_interceptor(request, handler, max_retries=3, delay=1.0):

## 面试高频问题

### 架构设计类

**Q1: 这个项目的整体架构是怎样的？**
> 回答要点：分层架构——API 层（FastAPI）→ 服务层（RagAgentService / AIOpsService）→ Agent 层（LangGraph 状态图 + Planner/Executor/Replanner）→ 工具层（本地工具 + MCP 工具）→ 基础设施层（Milvus + DashScope LLM）。前端是纯静态 HTML/JS，通过 SSE 接收流式响应。

**Q2: 为什么选择 LangGraph 而不是自己写 Agent 循环？**
> 回答要点：LangGraph 提供了状态图抽象，天然支持复杂的分支/循环逻辑（如 Plan-Execute-Replan 的三节点状态机）；内置 checkpoint 机制（MemorySaver）方便实现会话持久化；`create_agent` 自动实现 ReAct 循环。自己写灵活性高但容易出错且代码量大。

**Q3: MCP 协议在这个项目中的作用是什么？如果不使用 MCP，你会怎么设计？**
> 回答要点：MCP 解耦了 Agent 和外部工具，使得新增数据源不需要改 Agent 代码（只需新增 MCP Server）。不使用 MCP 的话，需要在 Agent 代码中直接写 HTTP 调用/数据库查询，工具和 Agent 紧耦合，难以维护和扩展。

### Plan-Execute-Replan 深度问题

**Q4: 请详细解释 Plan-Execute-Replan 和 ReAct 的区别？为什么运维场景更适合前者？**
> 回答要点：
> - ReAct：思考→行动→观察→思考→...，适合单步决策，但容易在复杂多步骤任务中"迷路"（缺少全局规划）。
> - Plan-Execute-Replan：先制定完整计划 → 逐步执行 → 每步后评估是否调整计划。有全局视角，适合需要多步骤协调的故障诊断。
> - 运维场景：故障诊断往往有固定的排查套路（先看告警→查日志→看监控→定位根因→给方案），Plan-Execute-Replan 天然适合这种有结构的诊断流程。

**Q5: 如何防止 Agent 陷入无限循环？**
> 回答要点：三层防护——硬编码限制（≥8 步强制结束）、半硬性限制（≥5 步禁止 replan）、Prompt 引导（"信息足够就响应，不要追求完美"）。此外新步骤数不能超过原剩余步骤数，防止步骤膨胀。

**Q6: 如果 Executor 执行步骤失败（比如 MCP 工具返回了错误），Replanner 会如何处理？**
> 回答要点：失败的步骤仍然会被记录到 `past_steps`（返回 "执行失败: xxx"），Replanner 会看到这个失败记录并做出决策：如果已有足够信息则 respond；如果关键步骤失败可能会 replan 换一种方式；如果是不重要的步骤则 continue 跳过。

**Q7: 为什么 Planner 要先检索知识库再制定计划？**
> 回答要点：运维诊断有大量经验知识（"CPU 高通常先查 top 进程"、"OOM 要查 dmesg"），这些经验以 Markdown 文档存在知识库里。Planner 先检索相关经验作为上下文，能让 LLM 制定出更专业、更符合实际运维套路的诊断计划，而不是泛泛的通用步骤。

### RAG 相关

**Q8: 文档是怎么变成向量并支持检索的？**
> 回答要点：用户上传 Markdown/TXT → `MarkdownHeaderTextSplitter` 按标题分割 → `RecursiveCharacterTextSplitter` 按大小二次分割 → 合并过小片段 → `DashScopeEmbeddings`（text-embedding-v4，1024 维）向量化 → 存入 Milvus（biz collection，IVF_FLAT 索引 + L2 距离）。检索时用户 query 向量化 → Milvus 近似搜索 Top-K → 格式化上下文 → 注入 LLM prompt。

**Q9: 为什么选择 Milvus 而不是其他向量数据库（如 Chroma、Pinecone、Weaviate）？**
> 回答要点：Milvus 是云原生分布式向量数据库，支持十亿级向量检索，性能好；支持多种索引类型；社区活跃；LangChain 有原生集成。Chroma 更适合轻量级场景；Pinecone 是 SaaS 有网络延迟；Weaviate 也不错但有额外运维成本。

**Q10: RAG 中的 chunk_size 和 overlap 是怎么设置的？为什么？**
> 回答要点：chunk_size=800、overlap=100、二次分割加倍。800 字符大约对应一段完整的运维知识段落；overlap=100 防止关键信息在分割边界被截断；Markdown 先按标题分割能保留文档结构（每个 chunk 知道自己属于哪个标题下），这在检索时能提供更好的上下文。

### 工程实践类

**Q11: 为什么 MCP 客户端要做重试？重试策略是怎样的？**
> 回答要点：网络抖动、MCP Server 临时不可用都可能导致工具调用失败。使用指数退避重试（1s→2s→4s），最多 3 次。通过拦截器模式实现，不侵入业务代码。重试全部失败后返回错误结果而不是抛异常，保证 Agent 能继续运行（降级处理）。

**Q12: 如何处理 LLM 调用失败的情况？**
> 回答要点：多处兜底——MCP 工具加载失败降级为仅本地工具；Planner 生成计划失败返回默认三步骤（收集信息→分析→报告）；Replanner 生成响应失败返回降级报告（包含原始任务和已执行步骤的摘要）；API 层统一 try/except 返回错误格式。

**Q13: 流式响应是怎么实现的？为什么要用 SSE 而不是 WebSocket？**
> 回答要点：FastAPI 的 `EventSourceResponse` + LangGraph 的 `astream(stream_mode="messages")`。SSE 比 WebSocket 更简单（单向推送即可，不需要双向通信），浏览器原生支持 `EventSource` API，自动重连。对于"LLM 推送给前端"这个场景，SSE 是最合适的。

**Q14: 如果要你把这个项目部署到生产环境，需要做哪些改进？**
> 回答要点：
> 1. MemorySaver → PostgresSaver/SqliteSaver（持久化会话，支持服务重启不丢失）
> 2. MCP 工具的 Mock 数据 → 接入真实的 CLS SDK 和监控系统
> 3. 增加认证鉴权（API Key / JWT）
> 4. CORS 限制具体域名而非 `*`
> 5. 增加请求限流（防止滥用）
> 6. 日志级别调为 WARNING（生产不应输出 DEBUG）
> 7. 增加监控告警（服务本身的健康度）
> 8. 考虑 LLM 调用的缓存（相同问题不重复调用 LLM）
> 9. 增加并发控制（LangGraph 的 checkpointer 在并发下需要考虑线程安全）

### 开放性问题

**Q15: 如果你要优化这个项目，你会从哪些方面入手？**
> 回答要点（选 2-3 个展开）：
> 1. **工具调用效率**：Planner/Executor/Replanner 各自独立获取 MCP 工具，可以改为全局共享，减少初始化开销
> 2. **Planner 的 Plan 质量**：可以引入 few-shot examples，让 LLM 学习优秀的诊断计划范例
> 3. **Replanner 的决策质量**：可以加入更多决策信号（如步骤结果的置信度、工具调用的耗时等）
> 4. **多 Agent 协作**：可以让多个 Agent 并行诊断不同告警，然后汇总
> 5. **工具描述的自动生成**：当前工具描述是手写的，如果工具很多会很痛苦

**Q16: 你觉得这个项目最大的技术挑战是什么？你是怎么解决的（或者你会怎么解决）？**
> 回答要点：最大的挑战是让 LLM 在有限的步骤内做出准确的诊断决策。解决方法包括：知识库提供经验参考（Planner 先检索再制定计划）；多层安全约束防止死循环；Prompt 精心设计（优先级口诀）；结构化输出（Pydantic 模型约束 LLM 的输出格式）。

**Q17: 如果要为这个项目增加一个新功能，比如支持图片/PDF 文档上传和问答，你需要改哪些地方？**
> 回答要点：
> 1. `app/api/file.py`：扩展 `ALLOWED_EXTENSIONS`，增加 PDF/图片解析逻辑
> 2. `app/services/document_splitter_service.py`：增加 PDF 分割器（如 PyPDFLoader）
> 3. 图片需要多模态 Embedding 模型（如 CLIP），或先用 OCR 提取文字再向量化
> 4. `app/config.py`：可能需要增加多模态模型配置
> 5. 前端 `app.js`：扩展文件类型验证
